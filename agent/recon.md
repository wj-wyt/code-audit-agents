# Phase 1 侦察 + Phase 1.5 业务理解

> 所有模式必读。定义 Phase 1（技术测绘）和 Phase 1.5（业务理解）的执行方法。

---

## Phase 1: 技术测绘

### 目标

在动手审计之前，建立对项目的完整心智模型：
- 这个系统是干什么的？
- 攻击者能从哪里进来？
- 系统里最有价值的东西是什么？

### Step 1: 项目结构理解

```
1. 枚举构建文件，理解模块划分
2. 识别技术栈和框架
3. 估算代码规模，确定审计范围
4. 分类模块:
   - 面向外部: 直接处理用户/网络输入的代码
   - 面向内部: 被其他模块调用的库/工具代码
   - 基础设施: 构建/部署/测试
5. 所有面向外部和面向内部的模块都是审计范围
   ⚠️ 不要只看 HTTP。任何 listen/bind/accept 都是入口
```

### Step 2: 攻击面枚举

不要只搜索 HTTP 端点。系统的攻击面包括：

| 类型 | 怎么找 | 为什么重要 |
|------|--------|----------|
| HTTP API | 路由注解/定义 | 最常见的入口 |
| 非 HTTP 端口 | listen/bind/accept 调用 | 开发者往往忽略这些端口的安全 |
| 文件解析 | 文件上传/导入接口 | 解析不可信数据是漏洞密集区 |
| 定时任务 | cron/scheduler 配置 | 可能处理外部数据但缺少校验 |
| 消息队列消费 | MQ consumer/subscriber | 消息内容可能被篡改 |
| CLI 接口 | 命令行参数处理 | setuid 程序尤其危险 |

### Step 3: 认证链理解

认证绕过放大所有其他漏洞。必须搞清楚：
- 认证在哪一层实现？Filter/中间件/注解？
- 认证链的顺序对不对？
- 哪些路径被豁免了？豁免的路径返回敏感数据吗？
- 不同 HTTP 方法的认证是否一致？

### Step 4: 高价值目标识别

在开始审计前，先找到系统里最值钱的东西：
- 密钥/凭证存在哪里？怎么生成的？谁能访问？
- 超级权限怎么获取？有没有内部认证机制？
- 最敏感的数据是什么？存储和传输加密了吗？
- 最危险的操作是什么？（RCE、文件写入、配置修改）

### Step 5: 快速排除（可选）

对明显不存在的攻击面快速排除：
- 项目没有 XML 解析 → 跳过 XXE
- 项目没有反序列化 → 跳过反序列化
- 用 Grep 快速确认，0 hits 则标记跳过

### Step 6: 静态分析预扫描（可选）

如果 semgrep/bandit/gosec 可用，跑一遍作为线索输入，但不作为发现。

### Phase 1 产出

```
[RECON]
项目规模 / 技术栈 / 项目类型
攻击面: HTTP 端点 + 非 HTTP 端口 + 文件解析 + 其他入口
认证链: 认证机制 → 豁免路径 → 潜在绕过点
高价值目标: 密钥/凭证/超级权限/敏感数据的位置
跳过列表: 确认不存在的攻击面
```

## Phase 1.5: 业务理解 ★

> 在技术测绘和漏洞扫描之间，理解"这个系统是干什么的"。

### 为什么重要

没有业务理解，你只能找到技术漏洞（SQL 注入、XSS）。
有了业务理解，你能找到逻辑漏洞（金额篡改、流程绕过、竞态条件）。

### Step 1: 业务域识别

读 README、数据模型、路由前缀，推断：
- 这是什么类型的系统？
- 核心实体是什么？（用户、订单、支付、文件...）
- 核心流程是什么？（注册→下单→支付→发货→退款）

### Step 2: 关键业务流程建模

对每个核心流程：
- 涉及哪些接口？按什么顺序调用？
- 哪些字段跟"钱"有关？（金额、余额、积分、优惠）
- 哪些字段跟"状态"有关？（订单状态、审批状态）
- 哪些操作不可逆？（支付、删除、权限变更）

### Step 3: 信任边界标注

对每个流程中的每个接口：
- 输入来源是什么？客户端 / 服务端计算 / 第三方回调？
- 哪些字段"不该由客户端控制"但可能被篡改？
- 哪些操作"应该有前置条件"但可能被跳过？

### Phase 1.5 执行条件

所有模式均完整执行 Phase 1.5。业务理解是发现逻辑漏洞的前提，不可简化。
