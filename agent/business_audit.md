# 业务逻辑审计 — 实战思维链

> D9 Agent 必读。本文件定义 business-driven 审计策略。
> 审计不是搜索，是提问。对每个业务流程，不是 Grep 关键词，而是逐步提问。

---

## 审计策略: business-driven

```
输入: Phase 1.5 的业务流程建模
方法: 按业务流程逐个审计，不是按维度/模式搜索
核心: 提问驱动，不是搜索驱动

执行序列:
1. 读 Phase 1.5 输出，获取核心业务流程列表
2. 对每个流程，按下方思维链逐步提问
3. 每个问题 → Read 对应代码 → 回答 → 记录发现
4. 流程审计完成后，做跨流程关联

D9 turns 弹性分配:
| 业务流程数 | 建议 turns |
|-----------|-----------|
| 1-3       | 20        |
| 4-6       | 30        |
| 7+        | 40（优先资金/权限流程） |

与 control-driven 的区别:
- control-driven: 枚举端点 → 检查有没有权限注解 → 没有就是漏洞
- business-driven: 理解业务流程 → 提问"这里应该检查什么" → 检查有没有 → 没有就是漏洞
- control-driven 能发现"缺少权限注解"
- business-driven 能发现"金额应该服务端计算但用了客户端传的值"
```

## 思维链: 支付/资金类流程

### Q1: 金额从哪来？
- Read Controller 的参数绑定 → amount 是请求参数还是服务端计算？
- Read Service 层 → 有没有从数据库/商品表重新取价格？
- 如果服务端直接用客户端传的 amount → **Critical（金额篡改）**

### Q2: 能不能付 0 元或负数？
- Read Service 层的金额校验 → 有没有 `amount > 0` 的检查？
- 优惠券/积分抵扣后能不能变成 0 或负数？
- 无校验 → **High（零元购/负数支付）**

### Q3: 支付回调验了什么？
- Read 回调接口 → 验签名？验金额一致性？验订单状态？
- 只验签名不验金额 = 可以用小额支付通过大额订单
- 金额不一致未校验 → **Critical（支付绕过）**

### Q4: 退款有没有上限？
- Read 退款接口 → refundAmount 和 paidAmount 的关系
- 能不能退款超过已支付金额？能不能重复退款？
- 无上限校验 → **High（超额退款）**

### Q5: 并发安全吗？
- 同一订单能不能同时发起两笔支付/退款？
- 余额扣减有没有用数据库行锁/乐观锁？
- Read SQL → `SELECT ... FOR UPDATE`? `@Version`?
- 无锁 → **High（竞态条件/重复扣款）**

### Q6: 优惠券/积分逻辑
- 优惠券能不能重复使用？领取有没有数量限制？
- 积分兑换有没有并发保护？
- 多种优惠能不能叠加超过商品价格？

## 思维链: 越权类流程

### Q1: 资源 ID 从哪来？
- URL 路径参数 `/order/{id}` → id 是用户可控的
- Read Service 层 → `findById(id)` 后有没有归属检查？
- 有没有校验 `order.userId == currentUser.id`？
- 无归属校验 + 端点可由普通用户访问 → **High（IDOR/水平越权）**

### Q2: 同一资源的不同操作权限一致吗？
- Read 同一个 Controller 的所有方法
- 列出所有 CRUD 方法的权限注解，做对比表:

| Controller | create | read | update | delete | export |
|-----------|--------|------|--------|--------|--------|
| OrderCtrl | ✅ | ✅ | ✅ | ❌ | ❌ |

- 不一致 → **High（垂直越权）**

### Q3: 批量操作安全吗？
- `POST /order/batch-delete {ids: [1,2,3]}`
- ids 里混入别人的订单 ID 会怎样？
- 服务端是逐个校验归属还是直接 `DELETE WHERE id IN (...)`？
- 无逐个校验 → **High（批量越权）**

### Q4: 数据导出范围
- export/download 端点的查询条件是否受限于当前用户/租户？
- 能否通过参数篡改导出其他租户数据？
- 无范围限制 → **Medium（数据泄露）**

## 思维链: 状态机类流程

### Q1: 状态转换有没有前置条件？
- Read `updateStatus` 方法 → 有没有检查当前状态？
- 能不能从"已取消"跳到"已发货"？
- 无前置条件检查 → **High（状态跳变）**

### Q2: 能不能跳步？
- 正常流程: 创建→支付→发货
- 能不能直接调发货接口跳过支付？
- 无流程校验 → **Critical（流程绕过）**

### Q3: 状态修改是原子的吗？
- Read SQL → 是在事务里吗？
- 两个请求同时修改状态会怎样？
- 无事务/锁 → **High（竞态条件）**

---

## 思维链: 注册/认证类流程

### Q1: 验证码能不能绕过？
- 验证码校验在哪一层？前端还是后端？
- 有没有过期时间？有没有次数限制？
- 删掉请求中的验证码字段会怎样？（服务端是否强制校验）
- 可绕过 → **High（验证码绕过）**

### Q2: 密码重置 token 可预测吗？
- Read token 生成逻辑 → UUID? 时间戳? 自增?
- token 有没有绑定用户？有没有过期？
- 可预测/无绑定 → **Critical（任意用户密码重置）**

### Q3: OAuth 回调安全吗？
- redirect_uri 校验严格吗？能不能改成攻击者的域名？
- state 参数有没有？能不能 CSRF？
- 校验不严 → **High（OAuth劫持）**

---

## 思维链: 多租户/多站点

### Q1: 查询条件是否强制包含租户标识？
- Read DAO/Repository 层 → SQL 是否包含 `AND tenant_id = ?`
- 能否通过篡改 tenantId 参数跨站操作？
- 无强制隔离 → **High（跨租户数据泄露）**

### Q2: 租户标识从哪来？
- 从 JWT/Session 中提取（安全）还是从请求参数中取（不安全）？
- 请求参数可控 → **Critical（租户隔离绕过）**

---

## 思维链: Mass Assignment

### Q1: 请求体直接绑定了什么？
- 搜索 `@RequestBody` / `@ModelAttribute` 绑定的实体类
- 实体类有没有 `@JsonIgnore` / `@JsonProperty(access=READ_ONLY)` 标注敏感字段？
- 是否使用 DTO 隔离（而非直接绑定 Entity）？

### Q2: 实体类有没有危险字段？
- role / isAdmin / status / siteId / balance / level
- 无隔离 + 实体含权限字段 → **High（Mass Assignment 提权）**

---

## 思维链: 文件上传/下载类流程

### Q1: 上传文件类型校验在哪一层？
- Read 上传接口 → 前端校验还是后端校验？
- 后端校验了什么？扩展名？MIME type？文件内容（magic bytes）？
- 仅前端校验或仅扩展名校验 → **High（任意文件上传）**

### Q2: 上传后文件存在哪？能不能直接访问执行？
- Read 存储逻辑 → 存到 Web 可访问目录？还是独立存储？
- 文件名是用户可控的还是服务端重命名？
- 可控文件名 + Web 目录 → **Critical（WebShell/getshell）**

### Q3: 下载接口有没有路径穿越？
- Read 下载接口 → 文件路径参数是用户可控的吗？
- 有没有 `../` 过滤？过滤是否可绕过（URL编码、双写等）？
- 路径可控 + 无有效过滤 → **High（任意文件读取）**

### Q4: 文件预览/解析安全吗？
- 如果有 Office/PDF/图片预览 → 解析引擎是什么？
- Excel 导入有没有 XXE？CSV 导入有没有公式注入？
- 图片处理用 ImageMagick → 检查 MVG/SVG payload
- 压缩包解压有没有 Zip Slip（`../` 路径覆盖）？

## 思维链: 审批/工作流类流程

### Q1: 审批流程能不能跳步？
- Read 审批接口 → 有没有检查当前节点是否是该审批人的待办？
- 能不能直接调用"终审通过"接口跳过中间审批节点？
- 无节点校验 → **High（审批流程绕过）**

### Q2: 能不能自己审批自己？
- Read 审批逻辑 → 有没有校验审批人 ≠ 申请人？
- 无校验 → **High（自审自批）**

### Q3: 审批状态能不能被直接修改？
- 有没有直接修改审批状态的接口（不经过审批流程）？
- 状态字段是否在 Mass Assignment 范围内？
- 可直接修改 → **Critical（审批绕过）**

### Q4: 已完成的审批能不能被篡改？
- 审批完成后，原始申请数据能不能被修改？
- 先提交正常申请→审批通过→修改申请内容 → **High（审后篡改）**

---

## 思维链: 数据隔离/权限继承

### Q1: 子账号权限是否正确继承？
- Read 子账号创建逻辑 → 子账号权限从哪来？
- 子账号能不能拥有超过父账号的权限？
- 无继承校验 → **High（权限提升）**

### Q2: 部门/组织数据隔离
- Read 数据查询层 → 查询条件是否包含部门/组织 ID？
- A 部门用户能不能看到 B 部门的数据？
- 无部门隔离 → **High（横向数据泄露）**

### Q3: 角色降级后权限是否回收？
- 用户从管理员降级为普通用户后，已有的 Session/Token 是否失效？
- 缓存的权限数据是否更新？
- 未回收 → **Medium（权限残留）**

---

## 思维链: API 安全

### Q1: 批量查询有没有限制？
- Read 列表查询接口 → 有没有分页限制？`limit` 参数有没有上限？
- `GET /api/users?limit=999999` 能不能拉全量数据？
- 无分页上限 → **Medium（数据过度暴露）**

### Q2: 接口版本管理安全吗？
- 旧版本接口（/api/v1/）是否仍然可访问？
- 旧版本是否缺少新版本已修复的安全控制？
- Read 路由配置 → v1 和 v2 的鉴权差异
- 旧版本无鉴权 + 仍可访问 → **High（旧版本接口绕过）**

### Q3: 错误响应泄露了什么？
- Read 全局异常处理器 → 错误响应包含什么？
- 是否泄露堆栈信息、SQL 语句、内部路径、用户是否存在？
- 不同错误码是否可用于用户枚举？（"用户不存在" vs "密码错误"）
- 泄露敏感信息 → **Medium（信息泄露）**

### Q4: GraphQL 特有风险（如适用）
- 内省查询是否开放？`{ __schema { types { name } } }`
- 嵌套查询深度有没有限制？（深度 DoS）
- 别名批量查询有没有限制？`{ a1: user(id:1) { ... } a2: user(id:2) { ... } }`
- 内省开放 → **Medium（Schema 泄露）**
- 无深度限制 → **Medium（查询 DoS）**

## 思维链: 微服务/网关

### Q1: 服务间认证安全吗？
- Read 服务间调用代码 → 有没有 mTLS/JWT/API Key？
- 内部服务是否信任所有来自网关的请求？
- 能否绕过网关直接访问内部服务？
- 无服务间认证 → **High（内部服务未授权访问）**

### Q2: 网关路由能不能绕过？
- Read 网关路由配置 → 路径匹配规则是什么？
- `/api/admin` vs `/api/admin/` vs `/api//admin` → 路径标准化差异
- 网关和后端服务的路径解析是否一致？
- 路径标准化不一致 → **High（网关鉴权绕过）**

### Q3: 内部 API 是否外泄？
- Grep 内部服务端口/地址 → 是否暴露在公网？
- Swagger/OpenAPI 文档是否包含内部接口？
- 健康检查/监控端点是否可公网访问？
- 内部 API 外泄 → **High（攻击面扩大）**

### Q4: 请求头传递安全吗？
- 网关是否透传 `X-Forwarded-For`、`X-User-Id` 等头？
- 后端服务是否信任这些头而不二次验证？
- 攻击者能否伪造 `X-User-Id: admin` 绕过认证？
- 信任可伪造头 → **Critical（认证绕过）**

---

## 跨流程关联（流程审计完成后执行）

完成所有流程的思维链审计后，检查:

1. **注册送积分 + 积分兑换** = 刷积分？
2. **邀请返佣 + 自推自** = 刷返佣？
3. **信息泄露 + 越权** = 获取他人数据后冒充操作？
4. **认证绕过 + 管理接口** = 未授权管理员操作？

---

## D9 Agent 输出格式

对每个业务流程，输出:
1. 流程图（接口调用顺序）
2. 信任边界标注（哪些输入不可信）
3. 逐问题回答（Q1-QN 的实际代码验证结果）
4. 发现列表（如有）

然后按标准 Agent 输出模板（HEADER → 发现表格 → SENTINEL）格式化最终输出。
