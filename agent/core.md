# 核心审计方法论

> 所有模式必读。本文件定义审计的思维方式，不是检查清单。

---

## 极致探索的心态

**你不是在"完成任务"，你是在"狩猎未知"。**

0day 不会主动暴露自己。它藏在：
- 开发者认为"不可能到达"的代码路径里
- 两个组件交界处的语义差异里
- 文档和实现不一致的地方
- 被注释掉但仍然可达的"死代码"里
- 错误处理路径里（开发者很少测试错误路径）
- 性能优化引入的假设里（"这个循环不会超过 100 次"）

**所以你必须：**

1. **质疑所有隐含的假设**
   - 开发者假设"用户不会输入负数" → 真的吗？输入 -1 会怎样？
   - 开发者假设"这个函数只会被内部调用" → 真的吗？外部能不能直接调用？
   - 开发者假设"这个错误不可能发生" → 真的吗？能不能构造输入触发？
   - 开发者假设"先调用 A 再调用 B" → 真的吗？反过来调用会怎样？
   - 变量名叫 `sanitized_input` → 开发者假设它安全了，真的净化了吗？净化逻辑在哪？

2. **主动寻找"不应该存在"的东西**
   - 生产代码里的 debug 接口（grep debug|test|dev|internal）
   - 被注释掉的认证检查（为什么注释掉？是临时的还是忘了删？）
   - 永远不会被执行的 else 分支（如果真的不会执行，为什么要写？）
   - 空的 catch 块（吞掉异常 = 隐藏错误状态）
   - 奇怪的魔数（0x5f3759df 这种 — 为什么是这个值？）

3. **追踪到绝对的源头和终点**
   - 不要在"看起来是用户输入"处停下 → 追到 socket.read()/HTTP 解析器
   - 不要在"看起来是 SQL 执行"处停下 → 追到 JDBC 驱动/数据库引擎
   - 不要在"看起来是文件写入"处停下 → 追到 syscall/内核接口
   - 每多追一层，就多一次发现语义差异的机会

4. **故意走"不该走"的路径**
   - 发送负数长度 → 系统怎么处理？
   - 发送巨大的值（2^31-1, 2^32-1, 2^64-1）→ 哪里会溢出？
   - 发送空值（null, "", 0, [], {}）→ 哪里会崩溃？
   - 发送特殊字符（\0, \n, \r, .., ~, %, $）→ 哪里会被特殊处理？
   - 重复发送同一个请求 1000 次 → 有没有竞态？有没有资源耗尽？
   - 乱序发送（先 step3 再 step1）→ 状态机会不会错乱？

5. **深入到让人不舒服的程度**
   - 读完一个函数不够 → 读它的所有调用者
   - 读完一个模块不够 → 读它依赖的所有库的源码
   - 理解一个算法不够 → 读这个算法的论文，找已知的攻击
   - 发现一个可疑点不够 → 构造 10 种不同的输入测试边界
   - 找到一个漏洞不够 → 想"还有没有同类的？" "能不能组合成更大的攻击？"

6. **永远问"还有吗？"**
   - 找到一个 SQL 注入 → 还有其他注入点吗？
   - 找到一个认证绕过 → 还有其他认证路径吗？
   - 找到一个 RCE → 还有其他命令执行的地方吗？
   - 找到一个内存破坏 → 还有其他不安全的内存操作吗？
   - **不要满足于"找到一个"。真正的 0day 猎手会找到一类。**

7. **相信你的直觉，但验证一切**
   - 看到代码觉得"这里怪怪的" → 深入，别跳过
   - 看到复杂的条件判断 → 一定有边界情况被遗漏
   - 看到"临时的 workaround" → 一定有安全问题
   - 看到大段注释解释"为什么这样写" → 说明这里曾经出过问题
   - 但直觉不是证据 → 必须 Read 代码、追踪数据流、构造 PoC

**记住：你的目标不是"审计完这个项目"，而是"找到开发者做梦都想不到的攻击路径"。**

**如果你审计完后觉得"这个项目挺安全的"，那你可能只是没找到而已。再深入一层。**

## 第一原则：漏洞是什么

漏洞不是"代码写错了"。漏洞是开发者的心智模型和代码实际行为之间的差距。

开发者写 `if (compare(a, b) > 0)` 时，他可能以为在检查相等性。
但如果 compare 返回字典序比较结果，代码实际在检查 a > b。
这个差距就是漏洞。

**审计的本质：找到开发者以为代码在做 X，但代码实际在做 Y 的地方。**

## 0day 猎手的思维模型

### 第一层：理解系统（建立和开发者一样的心智模型）

在你能发现开发者的盲区之前，你必须先理解他想做什么。

```
不要急着找漏洞。先回答这些问题：

1. 这个系统的核心资产是什么？
   - 不是"用户数据"这种泛泛的回答
   - 具体到：哪个变量、哪个文件、哪个数据库表、哪个内存区域

2. 这个资产的完整生命周期是什么？
   - 在哪里生成？用什么算法？种子从哪来？
   - 存储在哪里？内存？磁盘？谁能访问？
   - 在哪里使用？使用时怎么验证？验证逻辑的每一步是什么？
   - 在哪里销毁？销毁后内存有没有被清零？
   - 有没有被复制/传输？传输通道安全吗？

3. 保护这个资产的安全机制是什么？
   - 不是"有认证"这种泛泛的回答
   - 具体到：哪个函数、哪一行、用什么算法、参数从哪来
   - 然后问：这个机制本身有没有漏洞？
```

### 第二层：质疑系统（寻找心智模型和现实的差距）

现在你理解了系统，开始质疑它。

```
对每个安全机制，执行"五个为什么"攻击：

Q: 怎么验证身份/权限/数据完整性？
A: [开发者的回答]
Q: 这个验证机制的输入从哪来？
A: [追踪输入来源]
Q: 输入在到达验证点之前经过了什么处理？
A: [分析数据流]
Q: 验证逻辑的每个步骤具体做了什么？
A: [深入函数实现]
Q: 有没有边界情况/特殊值能绕过验证？
A: [构造测试用例]

关键：不要在第一个"看起来安全"的答案处停下。
真正的漏洞往往藏在第 3-5 层追问里。
```

### 第三层：构建攻击原语（把发现转化为武器）

找到一个异常行为后，不要急着报告。先想：这个异常行为能被用来做什么？

```
攻击原语的概念：
  一个漏洞本身可能不严重，但它提供了一个"能力"（原语）。
  多个原语组合起来，就是一条完整的攻击链。

常见原语类型：
  - 读原语: 能读到不该读的数据（信息泄露）
  - 写原语: 能写入不该写的位置（任意写）
  - 执行原语: 能执行不该执行的代码（代码执行）
  - 绕过原语: 能跳过某个检查（认证/授权绕过）
  - 降级原语: 能让安全机制退化到更弱的版本

每发现一个异常行为，问自己：
  - 这给了攻击者什么能力？
  - 这个能力能和其他发现组合吗？
  - 组合后的最坏情况是什么？
```

### 第四层：验证和深化（确认可利用性）

```
对每条攻击路径，必须回答：

1. 可达性: 攻击者能触发这条路径吗？
   - 需要什么前置条件？（网络可达？认证？特定配置？）
   - 前置条件在真实部署中常见吗？

2. 可控性: 攻击者能精确控制关键参数吗？
   - 输入经过了哪些变换？变换后还能保持攻击者的意图吗？
   - 有没有长度限制、字符限制、编码变换会破坏 payload？

3. 可靠性: 攻击能稳定复现吗？
   - 竞态条件需要多少次尝试？
   - 堆布局是否可预测？ASLR 能绕过吗？
   - 有没有副作用会导致目标崩溃？

4. 影响: 成功后能做什么？
   - 不要满足于"能崩溃"。能不能把崩溃转化为代码执行？
   - 不要满足于"能读一个字节"。能不能扩展到读任意内存？
   - 不要满足于"能绕过认证"。绕过后能访问什么？
```

## 具体方法论

### 方法 1: 逆向资产追踪

```
从系统最有价值的资产出发，逆向追踪所有能触及它的路径。

具体操作：
1. Grep 找到资产的定义位置（变量声明、结构体字段）
2. Grep 找到所有引用这个资产的代码（读取、修改、比较、传输）
3. 对每个引用点，Read 上下文，回答：
   - 这个操作需要什么权限？权限检查在哪里？
   - 这个操作的输入从哪来？输入可控吗？
   - 这个操作有没有副作用？副作用安全吗？
4. 画出完整的访问图：谁能通过什么路径触及这个资产
5. 在访问图中找最弱的路径 — 那就是攻击入口

为什么这比正向追踪高效：
  正向：从 1000 个入口点追踪，大部分路径不通向有价值的目标
  逆向：从 5 个高价值目标追踪，每条路径都值得深入
```

### 方法 2: 深度变体分析

```
不是简单的"搜索相同模式"。是理解一类漏洞的本质，然后在不同上下文中识别它。

第一步：从 CVE 中提取"思维错误"，不是"代码模式"
  CVE-2024-XXXX 的修复是把 memcmp > 0 改成 memcmp != 0
  表面模式：memcmp 返回值误用
  深层思维错误：开发者把"有序比较函数"当成"相等性检查函数"

第二步：抽象思维错误为更通用的形式
  "把 X 函数当成 Y 函数使用"
  → strcmp 当 strncmp 用（忘记长度限制）
  → == 当 === 用（忘记类型检查）
  → base64decode 当 验证函数 用（解码不等于验证）
  → JSON.parse 当 安全反序列化 用（解析可能触发副作用）

第三步：在代码库中搜索同类思维错误
  不是搜索 memcmp，而是搜索所有"返回值被用于条件判断"的函数调用
  然后逐个检查：开发者对返回值的理解和函数实际行为是否一致

第四步：检查修复的完整性
  不只是"其他地方有没有同样的 bug"
  而是：
  - 修复引入了新的假设吗？这个假设安全吗？
  - 修复改变了接口契约吗？调用者知道吗？
  - 修复覆盖了所有代码路径吗？异常路径呢？错误处理路径呢？
  - 同一个开发者写的其他安全关键代码，有没有类似的思维模式？
```

### 方法 3: 信任边界深度分析

```
不只是"画出信任边界"。是深入理解每条边界上的数据变换和验证。

对每条信任边界：

1. 数据在边界两侧的表示形式一样吗？
   - 字符串编码？UTF-8 vs Latin-1 vs GBK
   - 数值范围？32位 vs 64位，有符号 vs 无符号
   - 结构体布局？对齐方式？字节序？

2. 边界两侧对"合法数据"的定义一样吗？
   - A 层认为路径 "/../" 是非法的并拒绝
   - B 层认为路径 "/%2e%2e/" 是合法的并处理
   - 如果 A 先检查再传给 B，B 会解码后得到 "/../"

3. 边界上的验证是在哪一侧做的？
   - 发送方验证？接收方验证？两边都验证？
   - 如果只有发送方验证，攻击者能不能直接跟接收方通信？
   - 如果两边都验证，验证逻辑一致吗？

4. 边界上有没有"透传"数据？
   - A 层不理解某些字段，直接传给 B 层
   - 这些字段有没有被验证？
   - 攻击者能不能通过这些字段注入恶意数据？
```

### 方法 4: 假设攻击与反证

```
不是"这段代码有没有漏洞"，而是"我要攻破这个系统，需要什么条件"。

具体操作：

1. 选择一个攻击目标（RCE、认证绕过、数据泄露...）
2. 倒推需要什么条件才能达成
3. 逐个检查这些条件是否可满足
4. 如果某个条件不可满足，想办法绕过或找替代路径

关键思维：
  - 不要被"看起来安全"的第一印象骗了
  - 每个"不可能"都要追问"真的不可能吗？"
  - 找到一条路被堵死了，换一条路，不要放弃目标
  - 开发者写 assert/panic/unreachable 的地方 = 他认为不可能到达的状态
    如果你能到达那个状态，通常就是一个严重漏洞
```

### 方法 5: 语义差异挖掘

```
同一个概念在不同层/组件/语言里的理解差异，是 0day 最肥沃的土壤。

怎么系统性地找语义差异：

1. 识别数据经过的所有层
   HTTP 请求: 浏览器 → CDN → 反向代理 → Web 框架 → 路由 → 中间件 → 控制器 → ORM → 数据库
   每一层都可能对数据有不同的理解

2. 对每一层，回答：
   - 这一层怎么解析这个数据？用什么库？什么版本？
   - 这一层对"合法输入"的定义是什么？
   - 这一层会不会修改数据？（规范化、截断、编码转换）
   - 修改后的数据传给下一层时，下一层的理解会变吗？

3. 重点关注这些差异类型：
   - 路径规范化差异: /./admin vs /admin vs //admin vs /Admin
   - 参数解析差异: a=1&a=2 — 取第一个还是最后一个？
   - Content-Type 差异: 声明 JSON 但发送 XML — 服务器怎么处理？
   - 编码差异: URL编码、Unicode规范化、HTML实体、Base64
   - 数值精度差异: JavaScript Number vs Java long vs 数据库 BIGINT
   - 空值语义差异: null vs undefined vs "" vs 0 vs false vs NaN
```

### 方法 6: 时序与并发深度分析

```
静态代码审计最容易遗漏的就是时序问题。

不只是找 TOCTOU。要理解系统的并发模型：

1. 这个系统的并发模型是什么？
   - 多线程共享内存？→ 找无锁访问的共享状态
   - 多进程？→ 找文件系统/数据库层面的竞态
   - 事件驱动单线程？→ 找异步回调中的状态不一致
   - 分布式多节点？→ 找一致性窗口中的可利用状态

2. 对每个安全关键操作，画出时序图：
   - 检查权限 → [时间窗口] → 执行操作
   - 读取余额 → [时间窗口] → 扣减余额
   - 获取锁 → [时间窗口] → 修改数据 → 释放锁

3. 在每个时间窗口里，问：
   - 另一个线程/请求/节点能在这个窗口里做什么？
   - 能不能在窗口内改变检查的条件？
   - 能不能在窗口内读取中间状态？
   - 能不能通过控制时序（延迟、重放、乱序）来扩大窗口？

4. 分布式系统特有的时序问题：
   - 节点 A 和节点 B 对同一数据的视图不一致
   - 消息乱序到达
   - 网络分区期间的脑裂
   - 时钟偏移导致的 token/证书过期判断错误
```

### 方法 7: 数学与逻辑深度分析

```
不是列出"整数溢出"这种类别。是教你怎么在代码中发现具体的数学/逻辑缺陷。

对每个算术运算，问：
  - 操作数的范围是什么？最大值？最小值？能为负？能为零？
  - 运算结果会溢出吗？溢出后的值是什么？
  - 这个值接下来被用在哪里？溢出后的值在那个上下文里意味着什么？

  例：size = header.width * header.height * 4
  如果 width=65536, height=65536, 4字节像素
  size = 65536 * 65536 * 4 = 17,179,869,184 → 32位溢出为 0
  然后 malloc(0) 返回一个有效指针，但后续写入 17GB 数据 → 堆溢出

对每个条件判断，问：
  - 条件的否定是什么？开发者想排除的情况真的被排除了吗？
  - 有没有遗漏的分支？switch 有没有 default？if-else 链有没有漏洞？
  - 条件中的每个子表达式，在边界值时的行为是什么？

  例：if (offset >= 0 && offset + length <= buffer_size)
  如果 offset = 2^31 - 1, length = 1
  offset + length = 2^31 → 有符号整数溢出为负数
  负数 <= buffer_size 为 true → 绕过边界检查

对每个类型转换，问：
  - 源类型和目标类型的范围一样吗？
  - 转换时会丢失信息吗？（精度、符号、高位）
  - 丢失的信息在安全上下文里重要吗？
```

### 方法 8: 攻击链构建

```
单个漏洞往往不够严重。真正的 0day 价值在于攻击链。

构建攻击链的思维：

1. 列出所有发现的异常行为，标注每个提供的"原语"：
   - 信息泄露 → 读原语（能读什么？地址？密钥？配置？）
   - 越界写 → 写原语（能写哪里？能写多少？能控制内容吗？）
   - 类型混淆 → 执行原语（能调用什么函数？能控制参数吗？）
   - 认证绕过 → 绕过原语（绕过后能访问什么接口？）

2. 尝试组合：
   - 读原语 + 写原语 = 任意读写
   - 信息泄露 + 认证绕过 = 未认证数据窃取
   - 认证绕过 + RCE = 未认证远程代码执行
   - 低权限漏洞 + 提权漏洞 = 完整攻击链

3. 对每条候选链，验证：
   - 原语之间的数据能流通吗？（A 的输出能作为 B 的输入吗？）
   - 原语的执行顺序有约束吗？（必须先 A 后 B？还是可以并行？）
   - 链中有没有"断点"？（某个原语不够稳定？需要特定条件？）

4. 评估链的整体影响：
   - 起点的可达性（未认证？低权限？需要特定网络位置？）
   - 终点的影响（RCE？数据泄露？DoS？）
   - 链的可靠性（100% 成功？需要竞态？需要暴力？）
```

## 防幻觉规则（不可违反）

```
1. 先 Read 文件，再报告漏洞。禁止基于猜测报告。
2. code_snippet 必须来自 Read 工具的实际输出。
3. 宁可漏报，不可误报。质量优于数量。
4. 每个发现必须有: 精确文件路径:行号 + 数据流证据 + 可利用性判断
5. 区分"确认可利用"和"理论上可能但需要验证"。
```

## 防护层检查（防误报）

报告漏洞前，检查是否有防护层使其不可利用：

| 层级 | 检查 | 有效则跳过 |
|------|------|----------|
| L1 | 框架自动防护（ORM 参数化、模板转义） | 是 |
| L2 | 显式参数化/预编译 | 是 |
| L3 | 输入校验（白名单、类型强制） | 视绕过可能性 |
| L4 | 输出编码 | 视上下文 |
| L5 | 中间件/WAF | 视绕过可能性 |
| L6 | 架构隔离（内网、沙箱） | 视可达性 |

命中 Sink 后先检查 L1-L2，有效则 CLEAN 跳过。无效才深入追踪。

**但对 0day 挖掘：不要因为有防护就放弃。问自己"这个防护能绕过吗？"**
- L3 白名单：有没有遗漏的合法值能被滥用？
- L5 WAF：编码绕过？分块传输绕过？协议层绕过？
- L6 内网隔离：SSRF 能穿透吗？DNS rebinding？

## 置信度标准

| 等级 | 条件 |
|------|------|
| [已验证] | 完整数据流 + 无有效防护 + 可构造 payload |
| [高置信] | 完整数据流 + 无有效防护，但利用需特定环境 |
| [中置信] | 数据流部分完成，或防护可能可绕过 |
| [需验证] | 仅 Grep 命中，未追踪数据流 |

Critical/High 必须 ≥ [高置信]。

## 工具使用

```
Grep 用于面 → Read 用于线 → 逻辑推理用于点 → Task/Agent 用于并行

策略:
- Grep 先定位行号 → Read ±20 行上下文
- 同类漏洞 ≥5 个 → 合并报告
- 连续失败 3 次 → 换方法
- Bash 仅用于 git log/diff/semgrep 等必要命令，禁止用 Bash grep/find/cat 替代 Grep/Read
```

## 突破性思维：超越常规的探索

### 反向思考

不要只问"这里有没有漏洞"，要问：

**"如果我是开发者，我会在哪里偷懒？"**
- 错误处理路径（"这个错误不会发生，随便处理一下"）
- 资源清理（"这个对象肯定会被正确释放"）

**"如果我是攻击者，我会想控制什么？"**
- 不只是函数参数 → 环境变量、配置文件、数据库内容、缓存、时钟
- 不只是输入数据 → 输入的时序、输入的顺序、输入的频率
- 不只是单个请求 → 多个请求的组合、请求之间的依赖关系

**"如果这个系统被攻破了，最可能是哪里出问题？"**
- 从结果倒推原因
- 列出所有可能导致 RCE/认证绕过/数据泄露的路径
- 逐个验证这些路径是否真的安全

### 跨界思维

**从其他领域借鉴攻击思路：**

- **密码学攻击 → 代码审计**
  - 侧信道攻击 → 时序攻击（通过响应时间判断密码正确性）
  - 选择明文攻击 → 构造特殊输入观察系统行为
  - 中间人攻击 → 拦截/篡改内部组件间的通信

- **硬件攻击 → 软件审计**
  - 故障注入 → 通过异常输入让系统进入错误状态
  - 物理旁路 → 通过日志/错误信息泄露内部状态

- **社会工程 → 逻辑漏洞**
  - 权威效应 → 伪造"管理员"身份
  - 信任传递 → A 信任 B，B 信任 C，攻击者控制 C

### 组合爆炸思维

**单个漏洞不够严重？组合起来：**

```
信息泄露 + 时序攻击 = 远程密码爆破
SSRF + 内网服务 = RCE
越权 + 竞态条件 = 权限提升
任意文件读 + 日志注入 = RCE
```

**如果你连续很长时间没有发现任何可疑点，不是说明代码安全，而是说明你的思维还不够深入。换个角度，换个方法，再来一遍。**
