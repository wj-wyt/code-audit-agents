# 报告格式 + 严重度校准

> REPORT 阶段加载。定义报告结构、严重度校准、攻击链构建。

---

## 报告结构

```
1. 执行摘要（1页）── 审计范围、关键发现统计、最高风险总结
2. 漏洞统计表 ── Critical×N, High×N, Medium×N, Low×N
3. 漏洞详情（按等级降序）── Critical → High → Medium → Low
4. 攻击链分析 ── 多漏洞串联的端到端攻击路径
5. 修复优先级建议 ── 按业务影响排序
6. 正面发现 ── 项目做得好的安全实践
```

## 漏洞条目格式

每个漏洞必须包含:

```markdown
## [C-01] 漏洞标题

### 概述
简要描述漏洞性质和影响。

### 属性
| 项目 | 值 |
|------|-----|
| 严重程度 | Critical / High / Medium / Low |
| CVSS 3.1 | X.X |
| CWE | CWE-XXX |
| 置信度 | [已验证] / [高置信] / [中置信] / [需验证] |

### 受影响组件
- **文件**: `path/to/file.py:42`
- **函数**: `vulnerable_function()`

### 防护层状态
| 层级 | 防护类型 | 状态 |
|------|---------|------|
| L1 | 框架自动防护 | 无 / 有效 / 可绕过 |
| L2 | 参数化/预编译 | 无 / 有效 / 可绕过 |
| L3 | 输入校验 | 无 / 有效 / 可绕过 |
| L4 | 输出编码 | 无 / 有效 / 可绕过 |
| L5 | 中间件/过滤器 | 无 / 有效 / 可绕过 |
| L6 | 架构隔离 | 无 / 有效 / 可绕过 |

> Agent 必须在报告前完成防护层检查。未填写防护层状态的发现视为不完整。

### 漏洞代码
[关键代码片段，≤10行]

### 数据流
Source → Transform → Sink（完整路径）

### 攻击向量
描述攻击者如何利用此漏洞。

### PoC

### 修复建议
[修复代码示例]

### 参考
- CWE-XXX
```

## 报告质量标准

| 标准 | 要求 |
|------|------|
| 可定位 | 每个漏洞有精确的文件路径和行号 |
| 可复现 | 提供足够信息让开发者复现 |
| 可修复 | 给出具体的代码修复方案 |
| 无误报 | 每个漏洞都经过数据流验证 |
| 完整分析 | 说明完整利用路径和影响 |

## 严重度校准

> 多个 Agent 独立评级 → 同类漏洞可能被不同 Agent 评为不同等级。
> 主线程在合并去重后、写报告前执行校准。

### 决策树评分

对每个 finding:

| 维度 | 加分 |
|------|------|
| 可达性 | 未认证可达(+2) / 低权限(+1) / 管理员(+0) |
| 影响 | RCE或全库(+3) / 部分数据(+2) / 信息收集(+1) |
| 利用复杂度 | 单请求(+0) / 多步骤(-1) / 特定环境(-2) |
| 防护绕过 | 无防护(+0) / 可绕过(+0) / 需额外条件(-1) |

score = 可达性 + 影响 + 利用复杂度 + 防护绕过

| Score | 等级 |
|-------|------|
| ≥ 5 | Critical |
| 3-4 | High |
| 1-2 | Medium |
| ≤ 0 | Low |

### 校准规则

- 决策树等级 ≠ Agent 原始等级:
  - 差 1 级 → 取决策树等级
  - 差 ≥ 2 级 → 标记"等级争议"，报告中说明两种评估理由
- 同类漏洞统一等级: 同一 Sink 类别的多个实例 → 取最高实例等级

### 编号体系

| 前缀 | 含义 |
|------|------|
| C-XX | Critical |
| H-XX | High |
| M-XX | Medium |
| L-XX | Low |

## 攻击链自动构建

> 严重度校准后、写报告前执行。

### 构建流程

1. 列出所有 Critical/High 发现，标注:
   - 前置条件: 需认证(Y/N)? 需特定权限?
   - 利用结果: 信息泄露/RCE/权限提升/文件读写?

2. 自动匹配候选链:
   发现A的"利用结果" 满足 发现B的"前置条件" → 候选链 A→B
   例: 认证绕过(A) → 需认证的RCE(B) = A→B 链

3. 对每条候选链:
   - 验证数据流连通性
   - 给出组合等级（按链末端影响+链起点可达性重评）
   - 每条链最多 3 层延伸

4. 在报告"攻击链分析"章节输出，与独立漏洞列表分离

### 攻击链对等级的影响

- 漏洞A(认证绕过) + 漏洞B(需认证的RCE) → 漏洞B 按"未认证可达"重评
- 编号等级 = 独立等级（假设攻击者无其他漏洞）
- 攻击链部分单独给出"组合等级"

## 跨 Agent 发现冲突解决

同一漏洞被多个 Agent 从不同角度发现时:
- 取最高严重度等级
- 合并所有数据流证据
- 保留最完整的代码引用和行号
- 若攻击路径不同 → 视为同一 root cause 的不同利用路径

判定"同一漏洞": 同文件±20行 + 同 Sink 类型 + 同 root cause

## 发现去重规则

满足任一即为重复:
1. 同文件 + 同行号
2. 同文件 + 同漏洞类型 + 行号相差 < 10
3. 同文件 + 描述相似度 > 80%

合并策略: 保留更详细描述 + 更高等级 + 合并代码片段
