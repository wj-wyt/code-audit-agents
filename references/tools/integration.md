# 工具集成指南

> 工具是辅助，不是替代。工具输出是线索，不是发现。

---

## 核心原则

**工具的作用：**
- 快速定位可疑点（Sink、配置、依赖）
- 提供初始线索，缩小搜索范围
- 验证假设（"这个项目用了 X 框架吗？"）

**工具不能做的：**
- 理解业务逻辑
- 追踪跨文件数据流
- 判断漏洞的真实影响
- 发现项目特有的自定义协议/状态机漏洞

**所以：工具跑完后，必须 Read 代码验证。工具报告的不是漏洞，是"值得深入的线索"。**

---

## 可选工具

### 静态分析

```bash
# Semgrep（通用，支持多语言）
semgrep scan --config auto --json <target_path>

# Bandit（Python）
bandit -r <target_path> -f json

# Gosec（Go）
gosec -fmt=json ./...

# CodeQL（跨文件数据流，需要先建库）
codeql database create <db> --language=<lang> --source-root=<path>
codeql database analyze <db> <lang>-security-extended --format=sarif-latest
```

### 依赖扫描

```bash
# Python
pip-audit --format json

# Node.js
npm audit --json

# Go
go list -json -m all | nancy sleuth

# Java
mvn dependency-check:check
```

## 使用策略

**Phase 1 侦察后（可选）：**
1. 识别语言和框架
2. 选择对应的工具
3. 并行运行：静态分析 + 依赖扫描
4. 解析输出，生成"热点文件列表"
5. 优先审计热点文件，但必须 Read 验证

**不要：**
- 盲目信任工具输出
- 只审计工具报告的地方
- 把工具发现直接当成漏洞报告

**要：**
- 把工具输出当成"这些地方值得看看"的提示
- 对每个提示，Read 代码 → 理解上下文 → 追踪数据流 → 判断可利用性
- 花更多时间在工具找不到的地方（业务逻辑、自定义协议、时序问题）

---

## 工具的局限性

**工具找不到的漏洞（需要深度思考）：**
- 业务逻辑漏洞（支付绕过、状态机违规）
- 自定义协议的认证缺陷
- 时序/竞态条件
- 语义差异（路径解析差异、编码差异）
- 数学/逻辑缺陷（整数溢出、比较逻辑错误）
- 不完整修复的变体
- 开发者的隐式假设被打破

**真正的 0day 往往不在任何工具的规则库里。**

---

## 建议

对大多数项目，直接用 core.md 的方法论审计，不依赖工具。

只在这些情况下考虑用工具：
- 项目超大（100万行+），需要快速缩小范围
- 依赖很多，想快速检查已知 CVE
- 想验证"这个项目有没有用 X 危险函数"

工具节省的时间，应该用来做更深的思考，而不是审计更多表面问题。
